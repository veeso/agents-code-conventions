# Custom Rust Guidelines (X- prefix)

Custom extensions to the Microsoft Rust guidelines.
These conventions use the `X-` prefix to distinguish them from upstream `M-` conventions.

This file is NOT modified by the fetch-conventions script.

---


## Use `module_name.rs` Instead of `mod.rs` (X-NO-MOD-RS) { #X-NO-MOD-RS }

<why>To improve file navigation and avoid ambiguous tab names in editors.</why>
<version>1.0</version>

When organizing modules as directories, prefer the modern `module_name.rs` style over the legacy `module_name/mod.rs` style.

**Do:**

```
src/
├── lib.rs
├── network.rs          # Contains `mod connection;`
└── network/
    └── connection.rs
```

**Don't:**

```
src/
├── lib.rs
└── network/
    ├── mod.rs           # Avoid this pattern
    └── connection.rs
```

The `module_name.rs` style was stabilized in Rust 2018 edition. It provides better editor tab clarity
(seeing `network.rs` vs multiple `mod.rs` tabs) and is the recommended modern Rust convention.

## Prefer Let Chains Over Nested `if let` (X-LET-CHAINS) { #X-LET-CHAINS }

<why>Nested `if let` blocks increase indentation depth and obscure the logical relationship between conditions. Let chains express compound conditions flatly.</why>
<version>1.0</version>

When multiple conditions must hold simultaneously — especially combinations of boolean checks and pattern matches — prefer `let` chains (stabilized in Rust 1.87.0) over nested `if let` / `if` blocks.

**Do:**

```rust
if let Some(user) = get_user(id)
    && user.is_active()
    && let Some(email) = user.email()
{
    send_notification(email);
}
```

**Don't:**

```rust
if let Some(user) = get_user(id) {
    if user.is_active() {
        if let Some(email) = user.email() {
            send_notification(email);
        }
    }
}
```

When only one condition is checked, a plain `if let` is fine — this convention targets chains of two or more conditions that would otherwise nest.

For fallible chains where you need to propagate errors or return early, also consider the `?` operator or combinators like `.and_then()` instead of nesting.

## Prefer Iterator Combinators Over Nested Loops (X-ITER-COMBINATORS) { #X-ITER-COMBINATORS }

<why>Nested imperative loops with inner conditionals are harder to follow and more error-prone. Iterator combinators express data transformations declaratively and compose naturally.</why>
<version>1.0</version>

When processing collections, prefer iterator combinators (`.filter()`, `.map()`, `.filter_map()`, `.flat_map()`, `.fold()`, etc.) over imperative `for` loops that contain nested conditions or inner loops.

**Do:**

```rust
let active_emails: Vec<_> = users
    .iter()
    .filter(|u| u.is_active())
    .filter_map(|u| u.email())
    .collect();
```

**Don't:**

```rust
let mut active_emails = Vec::new();
for user in &users {
    if user.is_active() {
        if let Some(email) = user.email() {
            active_emails.push(email);
        }
    }
}
```

**Do** (flattening nested collections):

```rust
let all_tags: Vec<_> = articles
    .iter()
    .flat_map(|a| a.tags())
    .collect();
```

**Don't:**

```rust
let mut all_tags = Vec::new();
for article in &articles {
    for tag in article.tags() {
        all_tags.push(tag);
    }
}
```

A plain `for` loop is still appropriate when:

- The loop body has complex control flow (`break`, `continue` with labels, early `return` with `?`).
- Side effects make the imperative form significantly clearer.
- Performance-critical code where the optimizer handles the loop form better (measure first).

## Flatten Nested `match` Arms (X-FLAT-MATCH) { #X-FLAT-MATCH }

<why>Deeply nested `match` expressions are hard to read and maintain. Flattening them with tuple patterns or guard clauses keeps the logic at one level of indentation.</why>
<version>1.0</version>

When matching on multiple values that would otherwise require nested `match` blocks, prefer matching on a tuple or using guard clauses to keep the arms flat.

**Do** (tuple match):

```rust
match (config.mode(), config.level()) {
    (Mode::Fast, Level::High) => run_optimized(),
    (Mode::Fast, _) => run_fast(),
    (Mode::Safe, Level::Low) => run_minimal(),
    (_, _) => run_default(),
}
```

**Don't:**

```rust
match config.mode() {
    Mode::Fast => match config.level() {
        Level::High => run_optimized(),
        _ => run_fast(),
    },
    Mode::Safe => match config.level() {
        Level::Low => run_minimal(),
        _ => run_default(),
    },
    _ => run_default(),
}
```

**Do** (guard clause):

```rust
match event {
    Event::Click(pos) if pos.is_in_bounds(area) => handle_click(pos),
    Event::Click(_) => ignore(),
    Event::Key(k) if k.is_modifier() => update_modifiers(k),
    Event::Key(k) => handle_key(k),
}
```

**Don't:**

```rust
match event {
    Event::Click(pos) => {
        if pos.is_in_bounds(area) {
            handle_click(pos)
        } else {
            ignore()
        }
    }
    Event::Key(k) => {
        if k.is_modifier() {
            update_modifiers(k)
        } else {
            handle_key(k)
        }
    }
}
```

Nested `match` is acceptable when the inner match depends on bindings from the outer match that make tuple construction awkward, or when the inner arms are numerous enough that inlining them would bloat the outer match.

## Prefer `where` Clauses for Generic Bounds (X-WHERE-CLAUSE) { #X-WHERE-CLAUSE }

<why>Placing trait bounds in a `where` clause keeps function signatures readable, separates the parameter list from the constraints, and scales better as bounds grow.</why>
<version>1.0</version>

When a function, method, struct, enum, trait, or type alias accepts generic parameters with trait bounds, prefer expressing those bounds in a `where` clause rather than inline on the type parameter.

**Do:**

```rust
fn process<T>(items: &[T]) -> String
where
    T: Display + Debug,
{
    // ...
}

struct Cache<K, V>
where
    K: Eq + Hash,
    V: Clone,
{
    map: HashMap<K, V>,
}

impl<T> MyTrait for Wrapper<T>
where
    T: Send + Sync + 'static,
{
    // ...
}
```

**Don't:**

```rust
fn process<T: Display + Debug>(items: &[T]) -> String {
    // ...
}

struct Cache<K: Eq + Hash, V: Clone> {
    map: HashMap<K, V>,
}

impl<T: Send + Sync + 'static> MyTrait for Wrapper<T> {
    // ...
}
```

A single, simple bound on a single type parameter (e.g., `<T: Clone>`) is acceptable inline when it keeps the signature short and readable. Once any of these conditions hold, move bounds to a `where` clause:

- More than one trait bound on a type parameter.
- More than one bounded type parameter.
- Bounds include lifetimes, associated types, or complex paths (e.g., `T: Iterator<Item = u32>`).

## Document Feature Flags in `lib.rs` (X-LIB-FEATURE-FLAGS) { #X-LIB-FEATURE-FLAGS }

<why>Feature flags change a crate's public API and behavior. Documenting them in the crate-level docs gives users a single, discoverable reference for what each feature does and which are enabled by default.</why>
<version>1.0</version>

When a library crate defines one or more Cargo features, the `//!` crate-level documentation in `lib.rs` MUST contain a section titled `Feature flags` that documents every feature in a Markdown table.

The table MUST have one row per feature with these columns:

- `name` — the feature name.
- `description` — what the feature enables.
- `default` — `✔` if the feature is enabled by default, empty otherwise.

If the crate defines no features, this section MUST NOT exist.

**Do:**

```rust
//! # my-crate
//!
//! A crate that does something useful.
//!
//! ## Feature flags
//!
//! | name    | description                       | default |
//! |---------|-----------------------------------|---------|
//! | `std`   | Enable the standard library.      | ✔       |
//! | `serde` | Derive `Serialize`/`Deserialize`. |         |
//! | `async` | Enable the async API.             |         |
```

**Don't** (features exist but are undocumented):

```rust
//! # my-crate
//!
//! A crate that does something useful.
```

## Write `tracing` and `log` Messages Correctly (X-LOG-MESSAGES) { #X-LOG-MESSAGES }

<why>Log macros mix two independent mechanisms — structured key-value fields and format-string interpolation — and confusing them produces broken, duplicated, or literal-brace output. A common failure is emitting a structured field AND re-interpolating it in the message with double braces (`{{field}}`), which renders as literal `{field}` and silently corrupts every log line.</why>
<version>1.0</version>

This convention applies to the `tracing` macros (`trace!`, `debug!`, `info!`, `warn!`, `error!`, `event!`, `span!`) and the `log` macros (`trace!` … `error!`).

### The two mechanisms

A log macro call has two distinct parts:

1. **Structured fields** — `key = value` pairs written *before* the message string. Only `tracing` supports these. They are recorded as machine-readable metadata and do **not** appear inside the rendered message text.
   - `field = value` records the value directly (must be `Value`).
   - `field = %value` records `value` via its `Display` impl.
   - `field = ?value` records `value` via its `Debug` impl.
2. **The message** — a normal Rust format string (the same syntax as `format!`). Interpolation happens here via `{}`, `{name}`, or `{}`, `arg = expr` trailing arguments.

Fields go before the message; format args interpolate inside the message. They are separate — a structured field is NOT visible to the format string, and a format arg is NOT a structured field.

### Rules

- **Choose one mechanism per value.** Either record a value as a structured field OR interpolate it into the message — never both for the same value.
- **Never use `{{ }}` to interpolate.** In a format string `{{` and `}}` are escaped literal braces. `"config at {{config.path}}"` renders the literal text `config at {config.path}` — it does NOT substitute anything.
- **Structured fields do not print in the message.** If you record `config.path = %path` and want the path in human-readable text too, interpolate it explicitly with single braces — do not expect the field to appear on its own.
- **`log` has no structured fields.** With the `log` crate, only format-string interpolation exists; put every value inside the message with `{name}` capture or named trailing args.
- **No positional `{}` with bare trailing args.** `"This is {}!", user` is valid but unreadable — the placeholder gives no clue what it holds and the reader must count arguments. Capture the binding inline (`{user}`) or name the arg (`{name}", name = expr`).

### Correct

```rust
// Inline capture from scope (single braces capture the `pid` binding):
warn!("a router is already running (pid {pid}); stop it first or use a different --pidfile");

// Named trailing format argument:
info!("the path is {path}", path = db.path.display());

// Structured field, constant message — the field is metadata, not text:
info!(config.path = %path, "parsing router config");

// Self-documenting placeholders — captured binding or named arg:
info!("this is {user}!");
info!("this is {user}!", user = account.name());
```

### Wrong

```rust
// `{{config.path}}` is escaped literal braces: renders "parsing router config at {config.path}".
// It also duplicates the structured field. Double wrong.
info!(config.path = %path, "parsing router config at {{config.path}}");

// Same value both as a field and interpolated into the message — pick one.
info!(config.path = %path, "parsing router config at {path}", path = path.display());

// Positional placeholder with a bare trailing arg — unreadable, count-the-args.
info!("this is {}!", user);
```

### Fixing the wrong forms

If you want the path visible in the human-readable message, interpolate it once with single braces and drop the redundant field:

```rust
info!("parsing router config at {path}", path = path.display());
```

If you want it as machine-readable metadata, keep the field and a constant message:

```rust
info!(config.path = %path, "parsing router config");
```
